# Prueba 2 — Transaccionalidad

Demostrar con un escenario real por qué `@Transactional` es importante, comparando el comportamiento con y sin la
anotación.

---

## 🔄 Escenario de prueba

Usaremos el depósito porque involucra dos operaciones en la misma transacción:

````
1. Actualizar el saldo de la cuenta (UPDATE en accounts)
2. Registrar la transacción (INSERT en transactions) 
````

La idea es forzar un error entre esas dos operaciones para ver qué pasa con y sin `@Transactional`.

## ⚠️ Preparación

En el `AccountService` dentro del método `execute(String accountNumber, DepositCommand command)`, agrega esta línea
entre el `save(...)` de la cuenta y el `save(...)` de la transacción:

````java

@RequiredArgsConstructor
@Service
@Transactional(readOnly = true)
public class AccountService implements GetAccountsByCustomerUseCase, GetAccountByIdUseCase, GetAccountBalanceUseCase,
        OpenAccountUserCase, DepositUseCase, WithdrawUseCase, BlockAccountUseCase {
    // DEPOSIT
    // =========================================================
    @Override
    @Transactional
    public AccountResponse execute(String accountNumber, DepositCommand command) {
        // 1. Buscar cuenta y cliente
        Account account = this.findByAccountNumberOrThrow(accountNumber);
        Customer customer = this.findCustomerByIdOrThrow(account.getCustomerId());

        // 2. Ejecutar depósito — las reglas de negocio viven en el dominio
        Money amount = Money.of(command.amount(), account.getCurrency());
        account.deposit(amount);

        // 3. Persistir cuenta actualizada
        Account updatedAccount = this.accountRepositoryPort.save(account);

        // ⚠️ LÍNEA TEMPORAL PARA LA PRUEBA — simula un error entre operaciones
        if (true) throw new RuntimeException("Error simulado entre operaciones");
        //---------------------------------------------------------------------

        // 4. Registrar transacción exitosa
        Transaction transaction = Transaction.registerCompleted(
                this.accountApplicationHelper.generateReferenceNumber(),
                account.getId().value(),
                TransactionType.DEPOSIT,
                amount,
                command.description()
        );
        this.transactionRepositoryPort.save(transaction);

        return this.accountApplicationMapper.toResponse(updatedAccount, customer);
    }
}
````

## 🧪 Escenario B — CON `@Transactional`

### Paso 1: Descomenta el `@Transactional`

Descomentamos las dos anotaciones que habíamos comentado anteriormente:
`@Transactional` y `@Transactional(readOnly = true)`.

````java

@Service
@Transactional(readOnly = true) // ← Nuevamente habilitamos la anotación
public class AccountService implements GetAccountsByCustomerUseCase, GetAccountByIdUseCase, GetAccountBalanceUseCase,
        OpenAccountUserCase, DepositUseCase, WithdrawUseCase, BlockAccountUseCase {
    // DEPOSIT
    // =========================================================
    @Override
    @Transactional  // ← Nuevamente habilitamos la anotación
    public AccountResponse execute(String accountNumber, DepositCommand command) {
        // 1. Buscar cuenta y cliente
        Account account = this.findByAccountNumberOrThrow(accountNumber);
        Customer customer = this.findCustomerByIdOrThrow(account.getCustomerId());

        // 2. Ejecutar depósito — las reglas de negocio viven en el dominio
        Money amount = Money.of(command.amount(), account.getCurrency());
        account.deposit(amount);

        // 3. Persistir cuenta actualizada
        Account updatedAccount = this.accountRepositoryPort.save(account);

        // ⚠️ LÍNEA TEMPORAL PARA LA PRUEBA — simula un error entre operaciones
        if (true) throw new RuntimeException("Error simulado entre operaciones");
        //---------------------------------------------------------------------

        // 4. Registrar transacción exitosa

    }
}
````
